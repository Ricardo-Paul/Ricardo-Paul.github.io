The server socket is passive
The client socket is active, makes http requests.

Socket types

We refer to our socket as a stream socket, it is called as such because it allows processes to communicate using TCP, which is why the type is SOCK_STREAM. If we where to use UDP SOCK_DGRAM, we’d call it a datagram socket.

type qualifiers in C

const, volatile, restrict are type qualifiers in C
const as we know - means that the value wwon't
volatile

restrict type qualifier

restrict is a pointer qualifier (only used for pointesrs), it's used for optimization. if we write `int* restrict p1 = &a` here `p1` is a pointer to an int, restrict qualifier is applied to p1. This is telling the compiler,for the lifetime of this pointer only `p1` (the pointer name) wil be used to accessed the value `p1` is pointing to.

How accept works ?

accept(int socket, struct sockaddr *restrict address, socklen_t *restrict address_len);

  • first arg is the socket file descriptor
  • second arg: address is a pointer to the sockaddr struct, restrict is a type qualifier applied to pointer address

Accept takes a tcp socket (first TCP connection) but creates a new TCP socket that handles the connection to the client. The first socket remains open for new client connection requests.

Listening vs connected socket

  • The server starts by creating a new socket and bind it to a port, this socket is referred to as the listening socket.
  • When a client makes a connection request to the server, the operating system queueus the request.
  • accept is called on the listening socketd, it extracts a connection request from the queue and creates a new socket specifically for the client willing to connect, this is the connected socket or the client socket. This socket is destroyed as soon as the server has responded to the client.
  • The listening socket remains

When do we close the new socket (newly established TCP connection)

  • Write after responding ?
    In HTTP/1.0 that is the case, so if we are implementing HTTP/1.0 we need to close the TCP socket after writing a response to the client.
  • Do we not re-use the TCP connection ?
    In HTTP/1.1, persistent connection is introduced, if we are implementing it, we need to re-use the same TCP connection for future requests.

Reading from the socket

questions:

  • How is it possible to read data from a file descriptor, where does the file descriptor find this data ? why do we take the data from the fd and put it in the buffer ?

read() system call

  • read() sys call - read from a file descriptor to a buffer

write() system call

  • write to the file descriptor from a buffer
  • A file descriptor can represent different things, in our case it represents the socket. So read() is about reading data from the socket (data sent by the client) to a buffer. write() writes data to the socket (to send to the client)

lsof (list open files)

everything is a file in linux, including network connections.
this displays information about the process using port 8000
lsof -i TCP:8000

if we need to return the process ID we can append -t
lsof -i TCP:8000 -t

we can kill the process using its ID
sudo kill -9 $(lsof -i :8000 -tg)

Carriage Return Line Feed (CRLF) or \r\n

This is a combination of carriage return and new line that indicates the end of an http header.
It is recommended by the HTTP standard.
A wider space is required to separate the headers and the message body.. (\r\n\r\n)

A better mental image

The http client is reponsible to create a client socket (curl for example)
the operating system assigns an ip address and an EPHEMERAL port number to the socket
the server on its side will create a new socket on accepting the client request
the server can use getpeername on the client socket to retrieve the IP address and the EPHEMERAL port number assigned by the client OS.

There’s also getsockname which is used to get the server IP and port information.

Read the request line

The request line is the first line of the request made by an HTTP client
it looks something like this:GET / HTTP/1.0 which is method uri version

Remember when read() data from the client socket ? we put this data in a buffer, so the request line (included in the client reuqest )sent by the client is in the buffer.

listen() signature

int listen(sockfd, backlog)
backlog is a number that represents the size of the connection queue. SOMAXCONN is the value we’re using which is 128.
We know that the operating system queues the requests when they first come, and call accept() to remove a request from the queue. 128 is the limit number of connections to q can store. If a new request comes and the queue is already full, an ECONNREFUSED error will be sent to the client. The connection may succeed if the client retries it.
